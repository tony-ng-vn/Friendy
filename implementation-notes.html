Implementation Notes

Current State

  • Friendy is currently an iMessage-first relationship memory system. The canonical architecture source is docs/ai-system-architecture.md.
  • The active MVP detector scope is phone contacts plus calendar context. Fixture checks are deterministic, and real macOS Contacts/Calendar reads happen only through explicit local commands.
  • Added an optional SQLite runtime repository behind the existing relationship repository interface so the local checker and Spectrum/iMessage runtime can share pending candidates and memories across process lifetimes.
  • Kept in-memory repositories as the default for deterministic tests and fixture checks. SQLite is selected explicitly with FRIENDY_RUNTIME_STORE=sqlite and FRIENDY_SQLITE_PATH.
  • Used Node's built-in node:sqlite instead of adding a native dependency because the project already runs on Node 24.
  • Candidate Intake now owns queued contact candidate confirmation and ignore decisions for both deterministic and interpreted agents.
  • The SQLite relationship repository uses Node's built-in node:sqlite DatabaseSync, with indexed columns for query paths and raw JSON snapshots for evolving relationship objects.

Historical Starting Point

  • Initial build target is a local web product flow with mocked contact and calendar signals.
  • Native Contacts and Calendar integrations are intentionally deferred.
  • The app is implemented at the repository root because the Friendy repo was empty except for README.md.

Implementation Decisions

  • Added a repo-level AGENTS.md after the user asked for incremental commit and message-format instructions.
  • Used mocked contact deltas and a mocked Photon Residency Dinner calendar event to prove the agent loop before native iOS work.
  • Kept the UI chat-first, with side panels exposing event state, candidates, and saved memories for product flow clarity.
  • Added repo-owned docs for the product spec, product flow plan, handoff context, and Codex access setup so future sessions do not depend on external planning files.
  • Imported the original Superpowers spec and implementation plan into docs/superpowers/ so the planning trail is versioned with the repo.
  • Iteration 2 adds relationship-agent core modules under src/relationship/ so the current UI product flow remains stable while the agent architecture is built.
  • Changed TypeScript module resolution to Bundler so Vite can resolve Spectrum SDK package export subpaths such as spectrum-ts/providers/imessage.
  • The Spectrum adapter is a transport wrapper around the relationship agent core; the agent core remains usable through terminal tests without Spectrum credentials.
  • Installing spectrum-ts reported high-severity npm audit warnings in transitive dependencies; no automatic audit fix was run because it could make unrelated dependency changes during the MVP pass.
  • Added explicit .env.local/.env loading for standalone tsx agent scripts because Vite env loading does not apply to npm run agent:spectrum.
  • Expanded manual memory capture so the live agent accepts natural first-person messages like I met Amaya at Photon Residency II... instead of only command-style met Name, context inputs.
  • Added docs/agent-navigation.md to document the repo navigation strategy: keep REFERENCE.md as the index, keep AGENTS.md files short and scoped, and avoid turning always-loaded instructions into copied specs.
  • Added an LLM interpretation layer for Friendy messages. OpenRouter is used only to produce validated intent JSON; deterministic relationship tools still execute all memory writes, searches, ignores, and clarification behavior.
  • Kept Spectrum/iMessage as a transport adapter by moving interpreted execution into a testable Spectrum runtime helper that returns reply text plus compact interaction logs.
  • Contextual Memory Capture V2 adds a small conversation-context layer inside the interpreted agent so follow-up messages like also met Felix Ng... can inherit the active event and date context from recent turns.
  • Added chrono-node for natural-language date parsing. Friendy now stores both the raw date phrase and a normalized local date/date instant on relationship memories when a message contains temporal context.
  • Added a deterministic responseComposer boundary for user-facing save, search, no-match, clarification, and ignore replies. Search tools still select matches, while the composer prevents raw match reasons, score language, and placeholder labels such as manual contact from leaking into iMessage-style responses.
  • Updated deterministic memory search to score fields separately. Narrow searches now let role, project, school/class, alias, and specific context outrank generic shared event terms, while event-wide recall queries still return multiple people from the event.
  • Added candidateConfirmation.ts as the deterministic consent boundary for queued contacts. This keeps approval, ignore, and corrected-event handling in explicit tools instead of asking the LLM to directly mutate memory state.
  • Added list_candidate_event_matches so the agent can inspect ranked calendar guesses before confirming a candidate. Exact corrected event titles are preferred before substring matches, which prevents Photon Residency from accidentally keeping the more specific Photon Residency Dinner guess.
  • Changed the Spectrum runtime so a first inbound space can become the conversation identity when no explicit user ID exists. This removes the normal-onboarding dependency on a hardcoded owner phone while preserving explicit user IDs for tests and future signup flows.
  • Added src/relationship/evals/ as the trajectory-level agent evaluation boundary. The required eval suite is deterministic and local; optional repeated model-backed evals require both OPENROUTER_API_KEY and FRIENDY_EVAL_RUN_MODEL=1.
  • Added npm run eval:agent to report pass rate, intent accuracy, memory-write correctness, search recall@3, unsafe mutation count, hallucination count, clarification correctness, per-case assertions, and optional model-backed eval availability.
  • Added fixture contact snapshot diffing under src/relationship/ingestion/. The diff is method-centric: new normalized phone/email methods can create detections, while name-only edits and duplicate normalized methods are ignored.
  • Kept contact/calendar ingestion fixture-only for this goal. The calendar provider abstraction exists so real adapters can be added later, but tests, builds, evals, and npm run ingest:check do not read real calendars or Contacts.
  • Added sync_calendar_events to the relationship tools so ingestion can seed fixture events while event matching remains inside the existing repository/tool boundary.
  • Added an explicit macOS Contacts smoke command for manual integration checks. It only accepts Friendy-<number> names, creates or reuses one exact test contact, derives a deterministic phone number, and fails clearly outside macOS.
  • Removed product-flow wording that framed Friendy as something built for showing off instead of as the actual project. Renamed fixture commands and files to check:imessage-e2e, ingest:check, and *Flow/*Check names so future work treats them as product verification paths.
  • Added CHANGELOG.md as the project-level progress log from the initial scaffold through the current iMessage-first relationship-memory MVP.
  • Added the explicit local macOS contact/calendar checker as command-line tooling instead of a background watcher. This keeps real Contacts/Calendar reads user-triggered while proving the real sensor boundary.
  • Stored local checker snapshot state under .friendy/local-contact-snapshot.json and ignored .friendy/ because the file is machine-local state, not project source.
  • Kept local checker sending dry-run by default. Live Spectrum sending requires FRIENDY_LOCAL_CHECK_SEND=1 plus a target phone number so a local read cannot accidentally message the user.
  • Routed the local checker through the existing ingestion, repository, candidate queue, and prompt builder instead of creating a separate path for real Contacts/Calendar data.
  • Added candidateIntake.ts as the structured decision module for queued contact candidates. It selects, confirms, ignores, or reports ambiguity without composing final user-facing text.
  • Kept Candidate Intake behind relationship tools instead of direct repository writes. This preserves the tool boundary for future agent traces and keeps mutation behavior testable through the same interface the agents use.
  • Preserved ingestion compatibility by allowing bare confirmation only when exactly one pending candidate has event guesses. Multiple plausible event-matched candidates still require the user to choose.
  • Left durable shared state out of the Candidate Intake pass. The later runtime-store pass made pending candidates and relationship memory shareable across the local checker and live iMessage runtime.
  • For the SQLite repository pass, kept schema setup hand-written and scoped to the repository's current API. Users, calendar events, candidates, event matches, memories, and interactions each persist their full raw object while duplicating only the columns needed for filtering, sorting, and relationship identifier columns.
