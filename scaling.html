Friendy Scaling — Future Parking Lot

Purpose: Record scaling and architecture ideas that are correct to watch but not prioritized for MVP / current goals. Agents must not implement items here unless the user explicitly asks, or documented trigger metrics are met.

Canonical storage model (now): moment-centric, not profile-centric — PersonIdentity → many flat RelationshipMemory rows keyed by personId and memoryId. Grouping is a computed view (e.g. list_people, groupMemoriesByPerson), not a stored MemoryNode hierarchy.

See also: AGENTS.md (future-scaling rule), implementation-notes.html (decisions shipped now), docs/superpowers/specs/2026-05-23-identity-resolution-design.md.

Why flat memories beat a middle MemoryNode (for LLM tools)

  • Tool IDs: Models handle personId + memoryId better than personId + memoryNodeId + leaf id — fewer hierarchy hallucinations on mutate paths.
  • IR / search: Retrieval units should be episodic facts (high semantic density per row), not opaque containers that hide which moment matched.
  • Delete boundaries: Users say “forget that Sarah line” (leaf) or “forget Sarah” (person scope after disambiguation). A middle node duplicates personId without clarifying tool targets.
  • Verdict: Friendy is moment-centric. Keep storage flat; add summaries or joins only as background or computed layers — not as a structural tree the model manages.

Issue A — Context window stuffing (broad “tell me about Sarah”)

Problem: One person accumulates many flat rows (e.g. 50–500). Broad questions pull many fragments into context; synthesis gets expensive and noisy.

Deferred solution: Optional personSummary on PersonIdentity — markdown rollup updated asynchronously (cron / event listener). Models do not edit the summary via tools; episodic rows remain source of truth for search.

Trigger metrics (implement when any is true):

  • Typical power user has >30–50 active memories per personId
  • Broad person questions routinely exceed reply/token budget or quality drops in live logs
  • Product promises “everything you know about X” in one turn

Not the same as: duplicate display names / missing personId — that is identity (fix now).

Issue B — Memory fragmentation (repeated same fact)

Problem: Many rows state the same preference (“almond milk”) and dilute search ranking.

Partially now: Deterministic append paths (update_memory with mode: append); routing that avoids duplicate manual create for “is also …” phrasing.

Deferred solution: On add_memory / confirm, internal semantic check: if new fact maps to existing memoryId, patch or append instead of inserting a new row.

Trigger metrics:

  • Evals or logs show >N near-duplicate rows per person for the same semantic fact
  • Search top-K dominated by redundant preference/event lines

Issue C — Cross-person / group episodic memories

Problem: One moment involves multiple people (“dinner with Sarah and Alex at Photon”). Strict single personId per row forces duplication or weak group recall.

Deferred solution: MemoryParticipants join (or personIds[]) so one episodic row links multiple PersonIdentity records without copying text.

Trigger metrics:

  • Product requires saving/recalling group moments as first-class (not “search finds both names”)
  • Frequent user phrasing for multi-person capture with one shared context note

Anti-patterns (do not prioritize without user ask)

  • Stored hierarchy: displayName → MemoryNode → memories[]
  • Single giant profile blob per person as the only searchable document
  • Model-editable rolling summary without revision / grounding guardrails
  • Vector DB migration “because scale” before episodic + deterministic search limits bite
